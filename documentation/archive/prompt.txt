PROMPT 1.1 — MIDDLEWARE CLERK + RATE LIMITING
─────────────────────────────────────────────
Contexte : L'application Next.js (App Router) utilise Clerk pour l'auth et n'a AUCUN fichier middleware.ts à la racine de apps/web/src/. Les routes API sous apps/web/src/app/api/ ne sont protégées que manuellement via requireAdminAccess() ou requireAuthenticatedAccess() dans chaque route handler. Il n'y a aucun rate limiting.

Tâche :
1. Crée un fichier apps/web/src/middleware.ts utilisant clerkMiddleware() de @clerk/nextjs/server.
2. Configure le matcher pour protéger toutes les routes /api/* (sauf /api/health et /api/actions/map qui sont publiques) et toutes les routes /(app)/*.
3. Implémente un rate limiting basique en mémoire (Map<string, {count, timestamp}>) sur les routes POST /api/* : max 30 requêtes/minute par IP. Retourne 429 si dépassé.
4. Assure-toi que les routes publiques (/sign-in, /sign-up, /observatoire, la page d'accueil /) restent accessibles sans auth.

Fichiers à modifier : apps/web/src/middleware.ts (création), éventuellement next.config.ts pour le matcher.
Validation : npm run typecheck doit passer. Tester manuellement GET /api/health (200 sans auth) et POST /api/actions (401 sans auth).
Risques : vérifier que Clerk publishable key est bien exposée côté client via NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY dans env.ts.

──────────────────────────────────────────
Contexte : Les appels Supabase dans le codebase ne gèrent pas systématiquement les erreurs. Le pattern courant est : const { data } = await supabase.from(...).select(...) sans vérifier .error, ce qui provoque des data=null silencieux.

Tâche :
1. Recherche tous les appels supabase.from() dans apps/web/src/.
2. Identifie ceux qui n'ont pas de vérification de .error après la requête.
3. Ajoute une gestion d'erreur explicite : if (result.error) { console.error(...); return valeur_par_defaut_ou_throw }.
4. Ne masque jamais une erreur silencieusement. Log toujours avec console.error et un contexte suffisant (nom de la table, opération, userId si disponible).

Fichiers à modifier : tous les fichiers contenant des appels Supabase sans gestion d'erreur.
Validation : npm run typecheck. Grep "supabase.from" et vérifier que chaque occurrence a une gestion d'erreur à proximité.
Risques : ne pas transformer un comportement graceful-degradation existant en crash.
