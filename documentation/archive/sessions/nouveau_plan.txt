Vérifications prioritaires :
- `.gitignore`
- `.env.example`
- absence de secrets codés en dur
- éventuel `.dockerignore` si pertinent
- éventuels fichiers d'exemple de configuration sans secret
- documentation minimale sur la gestion des variables d'environnement
- tout autre fichier racine nécessaire à une base de projet propre et sûre

Pour éviter les fuites de secrets au push GitHub, les plus importants sont :

.gitignore
.env.example
.env.local non tracké
.env non tracké
.dockerignore si Docker
README.md avec section variables d'environnement
éventuellement un fichier de doc type documentation/securite/secrets.md
Point important

Le .gitignore protège seulement les fichiers non encore suivis.
Si un secret a déjà été commité une fois, l'ajouter ensuite au .gitignore ne suffit pas. Il faut alors :

retirer le fichier du suivi Git
faire tourner les secrets exposés
parfois nettoyer l'historique Git si l'exposition est réelle

Pour cette partie sécurité :
- signale précisément les risques détectés
- corrige seulement ce qui est justifié
- ne casse pas le fonctionnement du projet
- si un secret semble déjà exposé dans l'historique ou dans le code, signale-le explicitement
