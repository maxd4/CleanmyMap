DeMARCHE POUR LES SERVICES FUTURS (CL S API)
==================================================
Pour chaque service identifie  (Sentry, Posthog, Stripe, Pinecone, Resend, Upstash, UptimeRobot), suivre cette procedure :
1. Creer un compte sur la plateforme correspondante.
2. Generer une cle  API (Secret Key).
3. Ajouter la cle  dans le fichier .env.local de l'ordinateur local (sophi).
4. Ajouter le nom de la variable (sans la valeur) dans .env.example.
5. Mettre  à jour  pps/web/src/lib/env.ts pour que l'application valide la presence de la cle.
6. Verifier l' tat dans /api/services (depuis un compte admin).

Pour finaliser l'installation en production, assurez-vous que les variables suivantes sont configurées dans votre tableau de bord Vercel :

Monitoring : SENTRY_DSN, NEXT_PUBLIC_SENTRY_DSN, NEXT_PUBLIC_POSTHOG_KEY, NEXT_PUBLIC_POSTHOG_HOST.
Services : RESEND_API_KEY, CLERK_SECRET_KEY, SUPABASE_SERVICE_ROLE_KEY.


Supabase	NEXT_PUBLIC_SUPABASE_URL / SUPABASE_SERVICE_ROLE_KEY	Base de données (Admin/Server)
Clerk	NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY / CLERK_SECRET_KEY	Authentification & Profils
Sentry	NEXT_PUBLIC_SENTRY_DSN / SENTRY_DSN	Monitoring d'erreurs (Frontend & API)
PostHog	NEXT_PUBLIC_POSTHOG_KEY / NEXT_PUBLIC_POSTHOG_HOST	Analytics produit & Tracking serveur
Resend	RESEND_API_KEY / RESEND_FROM_EMAIL	Envoi d'emails transactionnels

Hydration failed because the server rendered HTML didn't match the client. As a result this tree will be regenerated on the client. This can happen if a SSR-ed Client Component used:

A server/client branch if (typeof window !== 'undefined').
Variable input such as Date.now() or Math.random() which changes each time it's called.
Date formatting in a user's locale which doesn't match the server.
External changing data without sending a snapshot of it along with the HTML.
Invalid HTML tag nesting.
It can also happen if the client has a browser extension installed which messes with the HTML before React loaded.

1. Modifications inclusives à appliquer (exemples principaux)
a) Navigation et sections
Remplacer « Page d’accueil » par « Accueil » (neutre, collectif)
« Agir », « Visualiser », « Impact », « Réseau », « Apprendre », « Piloter », « Guide & outils », restent inclusifs.

Tous les titres ou descriptions en double flexion si pertinent : Exemple français : « Membre/Participante·e », anglais : « Member/Participant ».
« Déclarer une action » → « Déclarer une action » (reste inclusif), mais attention à d’autres verbes ou instructions potentiellement marquées.
b) Filtres annuaire et formulaires
« Tous les acteurs » → « Tous les acteurs et actrices » ou « Toutes les personnes/structures ».
« Commerçants » → « Commerçant·e·s ».
« Associations », « Entreprises », « Collectifs » : inclusif.
Pour « Accueil »: rien à modifier.
Placeholder, label d’input et boutons : préférer les formulations neutres : « Nom/prénom », « Votre structure », « Contact », « Ajouter », « Rechercher ».
c) Badges, rôles, gamification
« Role bénévole » → « Bénévole »
« Role coordinateur » → « Coordination » ou « Coordinateur·rice »
« Role élu » → « Autorité locale » ou « Élu·e »
« Profil admin » → « Profil administration »
Pour tout rôle : recherche systématique d’équivalents épicènes ou usages de point médian.
Panel : « Badges d’interaction: mode, rôle, profil » → « Badges d’interaction : mode, rôle et profil » (fluidifier). Description reformulée pour éviter « coordinateurs », « élus », etc., remplacer par formulations collectives.
d) Feedback utilisateur & accessibilité
Bouton d’annulation : « Annuler le tracé » → « Annuler ».
Messages d’erreur : éviter « veuillez », préférer « Merci de », ou « Renseignez un champ ».
ARIA/alt : explicites, non genrés, tenir compte des déficiences visuelles (« Carte visuelle des actions », « Photo de profil »).
e) Exemples d’optimisation d’états et microcopies
« Niveau » (dans gamification) : « Niveau actuel » ou simplement « Niveau ».
« Badges gamification » → « Badges d’engagement ».
Balisage HTML : pour les boutons sans texte, aria-label explicite : « Ouvrir le menu principal », « Fermer la fenêtre », « Valider la sélection ».
2. Récapitulatif des changements effectués ou à effectuer
Réécriture des libellés, titres et descriptions pour neutralité de genre et ton accueillant : sections, navigation, badges, états utilisateurs.
Harmonisation des filtres et catégories avec point médian et termes englobants.
Accessibilité améliorée : aria-labels descriptifs, alt-text inclusifs, clarification des boutons et états.
Microcopies : instructions, placeholders, titres d’états vides ou d’erreur reformulés de façon épicène et claire.
Suppression des formulations genrées ou stéréotypées : usages de double flexion, point médian, ou collectif selon le contexte, maintien d’un ton professionnel et fluide.
