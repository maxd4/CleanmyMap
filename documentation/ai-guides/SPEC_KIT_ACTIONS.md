# CleanMyMap - actions a faire apres Spec Kit

Liste de travail derivee de la lecture de `github/spec-kit`.

## Actions prioritaires

- Formaliser un mini flux `spec -> plan -> tasks -> implement` pour les features majeures.
- Créer un emplacement standard pour les specs de fonctionnalite si le projet en veut un.
- Ajouter une quickstart de validation dans chaque plan important.
- Definir des criteres d'acceptation explicites avant de coder.
- Faire ressortir les contrats API, les modeles de donnees et les scenarios de test dans le plan.
- Reutiliser les presets/overrides locaux si une feature demande un comportement standardise.
- Documenter les limites des extensions avant de les adopter.
- Utiliser une version unique de l'outil Spec Kit pour eviter les variations de commande.
- Verifier que les scripts disponibles correspondent au shell cible, surtout sur Windows.
- Ajouter un point d'entree doc CleanMyMap qui explique quand passer de prompt direct a Spec Kit.

## Actions a moyen terme

- Harmoniser les artefacts spec/plan/tasks avec le reste de la doc CleanMyMap.
- Creer un exemple complet sur une feature reelle du projet.
- Ajouter un guide de review pour decider si une tache doit rester dans un prompt simple ou passer en Spec Kit.
- Definir quels types de changements necessitent un spec obligatoire.
