Objectif Global
Mettre en place un système durable de réduction de tokens et de continuité de contexte pour Codex, sans répéter les règles à chaque requête, avec un workflow réutilisable par session.

Contraintes Non Négociables

Une source d’instructions persistantes unique doit exister et être relue automatiquement.
Le contexte projet doit être stocké dans un fichier mémoire versionné et maintenu à jour.
Les prompts opérationnels doivent réutiliser un template court (pas de narratif).
Le setup doit fonctionner même entre sessions séparées.
Les règles “concis / diff-first / pas d’explications inutiles / code production” doivent être explicites et stables.
Étapes D’Implémentation (Base De Prompts Successifs)



PARTIE 1
Bloc 1 — Installer les instructions persistantes
But: Créer le “system prompt permanent” projet.
Zone de code probable: AGENTS.md (à créer), option locale ~/.codex/instructions.md.
Risque principal: Conflits entre règles locales et règles repo.
Prompt à exécuter: Crée AGENTS.md à la racine avec des règles globales courtes: concision, diff-only si possible, zéro répétition de contexte, priorité code production, style de réponse strict.

Bloc 2 — Créer la mémoire projet persistante
But: Centraliser architecture, conventions, décisions, limites connues.
Zone de code probable: project_context.md (à créer).
Risque principal: Fichier vite obsolète si non maintenu.
Prompt à exécuter: Crée project_context.md avec sections fixes: stack, architecture active, conventions, décisions ADR, zones sensibles, commandes de validation.

Bloc 3 — Définir le protocole d’ouverture de session
But: Forcer la relecture de contexte au début de chaque session.
Zone de code probable: documentation/du/archive/prompt_codex.txt ou nouveau documentation/du/session/session_bootstrap.txt.
Risque principal: Protocole ignoré en pratique.
Prompt à exécuter: Ajoute un bootstrap de 3 lignes maximum: "Lis AGENTS.md + project_context.md, confirme contraintes, propose plan en 5 blocs max".

Bloc 4 — Standardiser un template de prompt court
But: Éviter la réécriture manuelle des règles à chaque demande.
Zone de code probable: documentation/du/economie/economie_token_prompt_template.md (à créer).
Risque principal: Template trop générique donc faible qualité d’exécution.
Prompt à exécuter: Crée un template avec champs: Contexte minimal, Tâche, Contraintes, Sortie attendue, Vérification.

Bloc 5 — Mettre en place la mémoire de session continue
But: Garder un état inter-session court et exploitable (latest-session).
Zone de code probable: documentation/du/session/latest-session.md + script scripts/update_session_memory.mjs (à créer).
Risque principal: Inflation de taille et bruit inutile.
Prompt à exécuter: Crée latest-session.md structuré (done/in-progress/next/risks) et un script qui tronque à N lignes et met à jour uniquement ces sections.

Bloc 6 — Définir les règles de maintenance mémoire
But: Empêcher la dérive et les doublons entre AGENTS / project_context / latest-session.
Zone de code probable: documentation/repo-docs/ops/codex-memory-governance.md (à créer).
Risque principal: Triple source de vérité contradictoire.
Prompt à exécuter: Rédige la gouvernance: AGENTS=stable, project_context=semi-stable, latest-session=volatile, avec fréquence de mise à jour.

Bloc 7 — Intégrer dans les scripts de travail
But: Rendre le workflow exécutable en une commande.
Zone de code probable: package.json et README.md.
Risque principal: Processus défini mais jamais utilisé.
Prompt à exécuter: Ajoute scripts npm "session:bootstrap" et "session:close", et documente le cycle d’usage dans README.






PARTIE 2
Tu es en mode audit anti-token sur ce repo.

Objectif
Réduire fortement la consommation de tokens dans Codex sans dégrader maintenabilité, qualité produit, ni vitesse de delivery.

Méthode d’exécution (obligatoire)
1) Scanner le repo et identifier les sources réelles de gaspillage de contexte.
2) Prioriser uniquement les optimisations à fort ROI.
3) Proposer des refactors structurels minimaux (diffs ciblés, pas de réécriture globale).
4) Définir un workflow Codex strict pour limiter lecture/écriture inutile.
5) Proposer 3-5 automatisations concrètes applicables dans ce repo.
6) Estimer les gains (tokens, temps, itérations) avec hypothèses explicites.

Contraintes
- Maximum 10 constats.
- Chaque constat doit contenir: preuve fichier/ligne + impact + action.
- Favoriser quick wins d’abord.
- Aucune proposition vague (“améliorer la structure” sans plan concret).
- Ne pas casser le comportement existant.

Format de sortie (obligatoire)
A) Top sources de gaspillage (tableau)
Colonnes: Priorité | Preuve (fichier:ligne) | Type de gaspillage | Impact | Correctif concret

B) Plan d’optimisation priorisé
- Lot 1 (immédiat, <1h)
- Lot 2 (court terme, <1 jour)
- Lot 3 (structurel, >1 jour)

C) Workflow Codex anti-token (strict)
- Règles de lecture (ciblée, par fonction/bloc)
- Règles de sortie (courte, actionnable)
- Règles d’édition (diff-first, fichiers limités)

D) Automatisations recommandées
Pour chaque automation: but, implémentation, coût, gain attendu

E) Protocole anti-token step-by-step
Checklist exécutable avant/pendant/après chaque tâche (10 étapes max)

F) Gains estimés
- % réduction tokens prompt
- % réduction allers-retours
- % réduction temps de correction
(avec hypothèses)

Bloc 6 — Exécution finale + clôture opérationnelle
