# Resultat de l'audit pour identifier les principales sources potentielles de Vercel Edge Requests, ne fait que le tableau pour l'instant.

### Contexte :
- Le site a très peu d'utilisateurs réels.
- Pourtant plusieurs dizaines de milliers d'Edge Requests ont déjà été consommées.


### Les plus gros responsables
1. Le middleware/proxy lui-même sur les routes protégées et `__clerk`.
2. Le préchargement automatique des liens protégés dans la navigation globale.
3. Le bootstrap Clerk via `/__clerk`.
4. Le tracker de pageviews vers `/api/analytics/funnel`.

### Les optimisations les plus rentables
1. Réduire le périmètre du `matcher` dans [proxy.ts](C:/Users/sophi/Desktop/MAXENCE/business/CleanmyMap-main/apps/web/src/proxy.ts) au strict nécessaire.
2. Désactiver le prefetch sur les liens protégés visibles par défaut.
3. Alléger la dépendance à `__clerk` dans le shell global.
4. Batcher ou sampler les pageviews au lieu d’envoyer un POST par navigation.
5. Rendre le chat et les panneaux communauté plus paresseux au chargement.

### Plan d’action priorisé du plus urgent au moins urgent
1. Revalider le `matcher` du proxy et supprimer tout ce qui n’a pas besoin d’auth edge.
2. Passer en revue toutes les `Link` du shell global et couper le préfetch sur les cibles protégées.
3. Mesurer puis réduire l’usage de `/__clerk` dans le bootstrap auth.
4. Détacher le suivi de pageviews du chemin `Edge` si possible, ou au minimum le batcher.
5. Déporter les polls chat et les fetch automatiques communauté vers des déclenchements explicites.
6. Laisser Leaflet, PostHog, Sentry et Vercel Analytics de côté pour la facture Edge Requests: ce n’est pas le bon levier.
