# Audit Vercel Function Invocations

Contexte:
- Le site a très peu d'utilisateurs réels.
- Pourtant plus de 100 000 invocations Vercel ont déjà été consommées.

Ordre de priorité recommandé:
1. Réduire encore le middleware / proxy Clerk, car chaque requête Edge le paie immédiatement.
2. Garder le root layout public aussi statique que possible.
3. Retirer les auto-fetch et les revalidations au montage qui ne sont pas indispensables.
4. Supprimer les routes API qui ne font que relayer une lecture RLS-safe.
5. Réduire les pageviews analytics restants et le polling de supervision non visible.
6. Laisser les optimisations de niche sur la géo, la doc dynamique et l'observabilité externe pour la fin.

Note de gouvernance Clerk:
- Le shell public ne doit pas appeler `useUser` / `useAuth`.
- Les composants Clerk complets ne doivent se monter que sur les routes protégées.
- Si un composant partagé a besoin d'un badge ou d'un CTA de connexion, lui fournir une variante publique sans lecture auth.

Note de gouvernance API:
- Les listes relues au montage doivent avoir une borne claire et une cadence non live par défaut.
- Les suggestions saisie libre doivent attendre un seuil minimal avant de joindre le serveur.
- Pour la géo, privilégier le filtrage local avant toute requête distante, puis les caches courts sur les suggestions, le reverse geocoding et les spots lisibles.
- Les checklists synchronisées au fil de l'eau doivent debouncer leurs `POST` pour éviter un envoi par clic.
- Les endpoints de profil et de gamification les plus relus au montage (`/api/gamification/me`, `/api/gamification/leaderboard`, `/api/gamification/analytics/points`, `/api/users/profile/display-name-mode`) sont maintenant servis en cache court `private` ou via un loader dédié.
- Les routes qui ne font que relayer une lecture RLS-safe doivent être supprimées ou déplacées vers un livrable préchargé dès que l'usage le permet.
