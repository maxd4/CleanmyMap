Objectif Global
Mettre en place un système durable de réduction de tokens et de continuité de contexte pour Codex, sans répéter les règles à chaque requête, avec un workflow réutilisable par session.

Contraintes Non Négociables

Une source d’instructions persistantes unique doit exister et être relue automatiquement.
Le contexte projet doit être stocké dans un fichier mémoire versionné et maintenu à jour.
Les prompts opérationnels doivent réutiliser un template court (pas de narratif).
Le setup doit fonctionner même entre sessions séparées.
Les règles “concis / diff-first / pas d’explications inutiles / code production” doivent être explicites et stables.
Étapes D’Implémentation (Base De Prompts Successifs)



PARTIE 1
Bloc 1 — Installer les instructions persistantes
But: Créer le “system prompt permanent” projet.
Zone de code probable: AGENTS.md (à créer), option locale ~/.codex/instructions.md.
Risque principal: Conflits entre règles locales et règles repo.
Prompt à exécuter: Crée AGENTS.md à la racine avec des règles globales courtes: concision, diff-only si possible, zéro répétition de contexte, priorité code production, style de réponse strict.

Bloc 2 — Créer la mémoire projet persistante
But: Centraliser architecture, conventions, décisions, limites connues.
Zone de code probable: project_context.md (à créer).
Risque principal: Fichier vite obsolète si non maintenu.
Prompt à exécuter: Crée project_context.md avec sections fixes: stack, architecture active, conventions, décisions ADR, zones sensibles, commandes de validation.

Bloc 3 — Définir le protocole d’ouverture de session
But: Forcer la relecture de contexte au début de chaque session.
Zone de code probable: DU/prompt_codex.txt ou nouveau DU/session_bootstrap.txt.
Risque principal: Protocole ignoré en pratique.
Prompt à exécuter: Ajoute un bootstrap de 3 lignes maximum: "Lis AGENTS.md + project_context.md, confirme contraintes, propose plan en 5 blocs max".

Bloc 4 — Standardiser un template de prompt court
But: Éviter la réécriture manuelle des règles à chaque demande.
Zone de code probable: DU/economie_token_prompt_template.md (à créer).
Risque principal: Template trop générique donc faible qualité d’exécution.
Prompt à exécuter: Crée un template avec champs: Contexte minimal, Tâche, Contraintes, Sortie attendue, Vérification.

Bloc 5 — Mettre en place la mémoire de session continue
But: Garder un état inter-session court et exploitable (latest-session).
Zone de code probable: DU/latest-session.md + script scripts/update_session_memory.mjs (à créer).
Risque principal: Inflation de taille et bruit inutile.
Prompt à exécuter: Crée latest-session.md structuré (done/in-progress/next/risks) et un script qui tronque à N lignes et met à jour uniquement ces sections.

Bloc 6 — Définir les règles de maintenance mémoire
But: Empêcher la dérive et les doublons entre AGENTS / project_context / latest-session.
Zone de code probable: docs/ops/codex-memory-governance.md (à créer).
Risque principal: Triple source de vérité contradictoire.
Prompt à exécuter: Rédige la gouvernance: AGENTS=stable, project_context=semi-stable, latest-session=volatile, avec fréquence de mise à jour.

Bloc 7 — Intégrer dans les scripts de travail
But: Rendre le workflow exécutable en une commande.
Zone de code probable: package.json et README.md.
Risque principal: Processus défini mais jamais utilisé.
Prompt à exécuter: Ajoute scripts npm "session:bootstrap" et "session:close", et documente le cycle d’usage dans README.










PARTIE 2 
Audit anti-token
Usage: réduire rapidement les coûts de contexte et les itérations inutiles.

Prompt:
"Audit le repo pour réduire fortement la consommation de tokens dans Codex.
Objectif: obtenir un standard opérationnel anti-gaspillage sans nuire à la maintenance.

À produire:
1) Sources de gaspillage (preuves concrètes)
2) Optimisations à plus fort impact
3) Refactors de structure (fichiers trop gros, redondance, modularité)
4) Workflow Codex strict (récupération ciblée, lecture fonction/bloc, sorties courtes)
5) Automatisations utiles (cache, plafonds de contexte, compression inputs)
6) Gains estimés

Livrable obligatoire: protocole anti-token step by step
Contraintes: critique, concret, sans réécrire tout le repo.




PARTIE 3
Bloc 1 — Audit global rapide (détection macro)
But: Identifier les risques systémiques et établir une matrice de priorité unique.
Zone de code probable: apps/web/src/app, apps/web/src/lib, apps/web/src/components, apps/web/src/app/api.
Risque principal: Audit trop large et verbeux, inutilisable en exécution.
Prompt opératoire: Audit global rapide du repo. Donne uniquement: 1) 10 problèmes max 2) sévérité 3) preuve fichier/ligne 4) impact user/produit.

Bloc 2 — Audit par rubrique (couche principale)
But: Couvrir les 15 rubriques sans mélange de contexte.
Zone de code probable: apps/web/src/components/sections/rubriques, apps/web/src/app/(app)/sections/[sectionId]/page.tsx, apps/web/src/lib/sections-registry.ts.
Risque principal: Oublier des rubriques ou dupliquer les constats.
Prompt opératoire: Audit rubrique par rubrique (batch de 3 rubriques). Pour chaque rubrique: UX, bugs, cohérence, performance. Sortie: constats + actions correctives concrètes.

Bloc 3 — Audit transversal inter-rubriques
But: Détecter incohérences navigation/design/logique utilisateur entre rubriques.
Zone de code probable: apps/web/src/lib/navigation.ts, apps/web/src/lib/sections-registry.ts, apps/web/src/components/navigation/app-navigation.tsx, layouts (app).
Risque principal: Valider localement des pages qui cassent le parcours global.
Prompt opératoire: Analyse transversale navigation/design/CTA/permissions entre rubriques. Liste les collisions et régressions de parcours.

Bloc 4 — Correction Codex (critiques puis importants)
But: Appliquer les fixes en ordre de risque, avec changements regroupés par zone.
Zone de code probable: fichiers signalés blocs 1-3 (priorité auth/navigation/data/map).
Risque principal: Régression croisée (contrat data, navigation, auth).
Prompt opératoire: Applique les corrections critiques puis importantes, fichier par fichier, avec tests ciblés après chaque lot.

Bloc 5 — Passe “expert minimal”
But: Obtenir une revue finale d’angles morts avec output ultra court.
Zone de code probable: diff git complet + zones sensibles (authz.ts, proxy.ts, unified-source.ts, navigation.ts).
Risque principal: Sortie trop longue, faible ROI token.
Prompt opératoire: Review finale en 10 points max, sans explication longue, uniquement problèmes critiques restants + angle mort.

Bloc 6 — Exécution finale + clôture opérationnelle
But: Corriger les points restants, figer le protocole d’audit anti-token dans la doc DU.
Zone de code probable: fixes finaux + DU/economie_token.txt (append protocole exécutable) + doc ops.
Prompt opératoire: Applique les derniers correctifs.
